将以下需求拆解为可独立执行的子任务：

$ARGUMENTS

## 拆解流程

### 第一步：理解需求

- 分析需求的业务目标和用户价值
- 识别涉及的系统模块和技术领域
- 列出需要澄清的问题（如有，先列出再继续）

### 第二步：拆解为子任务

研发流程必须包含以下阶段，按顺序排列：

1. **原型设计**（必须）：技术方案文档 + 最小可运行原型
2. **▶ 原型审批门禁**（必须）：人工确认数据模型/API/技术选型
3. **功能实现**（审批通过后）：生产级代码 + 边界处理
4. **测试验证**（必须）：测试套件 + 验收确认
5. **▶ 合并审批门禁**（必须）：人工确认可合并

每个子任务必须满足：

1. 可以由一个 Agent 独立完成，不依赖其他未完成子任务的产出
2. 有明确的验收标准，Agent 可以自行判断是否完成
3. 预估复杂度合理（单个子任务不超过 8 小时 Agent 工作量）

每个子任务包含以下字段：

- **标题**：简明扼要
- **阶段**：原型设计 / 功能实现 / 测试验证
- **目标**：预期产出什么文件/功能/变更
- **验收标准**：具体的通过条件（可运行的测试、可观测的行为）
- **复杂度**：S（<1h）/ M（1-4h）/ L（4-8h）/ XL（>8h，需进一步拆分）
- **推荐引擎**：Cursor Long-running / Cursor Background / Claude Code Headless
- **依赖**：前置子任务编号（无依赖标注 `无`）

### 第三步：标注并行关系和审批门禁

- 无依赖关系的子任务标注为 `[PARALLEL]`
- 有依赖的标注为 `[SEQUENTIAL: 依赖任务编号]`
- 审批门禁标注为 `[GATE: 需人工审批]`
- 画出包含审批门禁的依赖图
