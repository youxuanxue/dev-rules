将以下需求拆解为可独立执行的子任务：

$ARGUMENTS

## 拆解流程

### 第一步：理解需求 + 聚焦过滤

- 分析需求的业务目标和用户价值
- 识别涉及的系统模块和技术领域
- **聚焦过滤（乔布斯原则）**：
  - 这个需求要解决的**一个核心场景**是什么？
  - 哪些功能点「很诱人但不是核心」？——明确列出「不做什么」及原因
  - 砍掉的功能和保留的功能同样重要，必须在拆解报告中说明
  - 判断标准：如果这个功能消失了，核心场景是否仍然完整？如果是，则暂缓
- 列出需要澄清的问题（如有，先列出再继续）；常规风险以上必须说明歧义如何消解：用户明确回答、现有规则约束、代码事实，还是审批基线决定

### Context Pack（仅在有助于减少误读时输出）

低风险任务不输出 Context Pack。常规风险以上、跨模块任务、或需要派发给其他 Agent 时，输出一个最小上下文包：

- 当前任务意图
- 相关规则或审批产物路径
- 相关 spec delta / Story / 测试入口（如有）
- 关键代码入口
- 本次验证命令

Context Pack 只放高信号引用，不粘贴大段文件内容；Memory 只能作为查找线索，不能覆盖当前代码事实。

### 第二步：先判定风险等级，再决定流程

风险分级、默认路径和高风险升级条件**直接遵循** `rules/product-dev.mdc`，不要在本命令里另写一套定义；这里只需给出本次需求的结论：

- `低风险`
- `常规风险`
- `高风险`

### 第三步：按风险决定交付路径

#### 默认路径（低风险 / 常规风险）

默认采用**单 PR**：

```text
单 PR → 实现/测试 → preflight → review → 人工确认
```

只输出为完成当前变更所需的最少子任务；不要为了流程完整性强行生成 docs PR、prototype PR、上线 PR。

- 低风险需求默认可直接在对话中给出精简拆解；除非用户要求留档，否则不要强制写 `docs/task-breakdown-*.md`
- 常规风险不默认写文档；只有当行为变化复杂到 PR summary 说不清，或 reviewer 需要先理解 intent 再看 code 时，才建议补 `docs/spec-delta-<slug>.md`
- spec 只写 what / why；plan 才写 how；tasks 只写执行单元、依赖和验收，不重新定义需求

#### 高风险路径

只有高风险需求才要求完整阶段：

1. **需求分析**：任务拆解 + 聚焦过滤
2. **原型设计**：`docs/approved/` + 最小可运行原型
3. **▶ 原型审批门禁**
4. **功能实现**
5. **测试验证**
6. **▶ 合并审批门禁**

### 第四步：拆解为子任务

每个子任务必须满足：

1. 可以由一个 Agent 独立完成，不依赖其他未完成子任务的产出
2. 有明确的验收标准，Agent 可以自行判断是否完成
3. 预估复杂度合理（单个子任务不超过 8 小时 Agent 工作量）

每个子任务包含以下字段：

- **标题**：简明扼要
- **阶段**：原型设计 / 功能实现 / 测试验证
- **目标**：预期产出什么文件/功能/变更
- **验收标准**：具体的通过条件（可运行的测试、可观测的行为）
- **复杂度**：S（<1h）/ M（1-4h）/ L（4-8h）/ XL（>8h，需进一步拆分）
- **引擎**：按下方路由规则自动决定（不是建议，是规则）
- **依赖**：前置子任务编号（无依赖标注 `无`）
- **PR 归属**：默认写 `同一 PR`；只有明确写出拆分原因时，才拆成多个 PR

### 引擎路由规则（必须遵守）

根据任务特征自动选择引擎。默认沿用用户当前工作环境；只有在明显更适合无人值守或长时运行时，才切换。


| #   | 条件                | 引擎                   | 原因            |
| --- | ----------------- | -------------------- | ------------- |
| 1   | 高风险，且实现跨度大或预计长时运行 | Cursor Long-running  | 需要深度工作与跨文件协调  |
| 2   | 纯文档、巡检、报告或批量化脚本任务 | Claude Code Headless | 更适合无人值守       |
| 3   | 其余情况              | 当前默认环境               | 保持路径单一，减少无谓调度 |


路由决策记录在每个子任务的 `引擎` 字段中，格式为 `引擎名称（规则 #N）`，例如 `Cursor Long-running（规则 #1）`。

### 专家团触发判断（默认不用）

专家团只服务高价值、可并行、边界清晰、需要多视角的任务。满足以下条件之一时才建议启用：

- 高风险变更，且影响架构 / 安全 / 数据 / 公共契约
- 大型 PR review，需要安全、测试、架构并行审查
- 根因不明的复杂 bug，需要竞争假设
- 跨前后端和测试的功能，且文件边界可拆
- 用户明确要求“专家团”或“多角色审查”

以下情况禁止为了形式启用专家团：低风险文档/命名/格式化、单点 bug fix、同一文件集中修改、需求未澄清、顺序依赖强。

若启用专家团，必须指定 Lead 合成结果；每个专家只负责独立 scope，并输出证据、风险、建议和置信度。

### 第五步：标注并行关系和审批门禁

- 无依赖关系的子任务标注为 `[PARALLEL]`
- 有依赖的标注为 `[SEQUENTIAL: 依赖任务编号]`
- 只有高风险路径中的审批门禁才标注为 `[GATE: 需人工审批]`
- 画出包含审批门禁的依赖图
